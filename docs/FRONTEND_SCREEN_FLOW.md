# 프론트엔드 화면 흐름 설계

## 1. 목적

Re:Start Quest의 MVP 화면은 취업 기능을 많이 보여주는 데 초점을 두지 않는다. 사용자가 오늘 다시 시작할 수 있는 작은 구직 행동을 받고, 실패했을 때 더 쉬운 행동으로 재설계한 뒤, 대시보드에서 다음 행동을 바로 확인하는 흐름을 우선한다.

핵심 화면 흐름은 다음 한 문장으로 검증한다.

```text
온보딩 -> 오늘의 퀘스트 생성 -> 실패 이유 입력 -> 더 쉬운 퀘스트 재설계 -> 대시보드 반영
```

## 2. MVP 화면 범위

| 화면 | 권장 라우트 | MVP 책임 | 제외할 것 |
|---|---|---|---|
| 로그인 / 회원가입 | `/login`, `/signup` | 사용자 식별, 인증 실패 안내 | 소셜 로그인 다중 연동 |
| 온보딩 | `/onboarding` | 희망 직무, 공백 기간, 이력서 상태, 오늘 에너지 입력 | 심리 진단, 의지 평가 |
| 오늘의 퀘스트 | `/quests/today` | 10~30분 단위 퀘스트 3개 확인, 완료/실패 진입 | 무한 추천 목록 |
| 실패 이유 입력 | `/quests/:questId/fail` 또는 모달 | 실패 이유 선택, 짧은 메모 입력, 재설계 요청 | 실패 점수화, 부정적 피드백 |
| 재설계 결과 | `/quests/:questId/redesign` 또는 단계형 패널 | 더 작은 행동 목록 확인, 오늘 할 행동으로 수락 | 복잡한 AI 대화창 |
| 대시보드 | `/dashboard` | 오늘 진행률, 재설계 기록, 다음 행동 표시 | 전체 취업 통계 대시보드 |
| 활동 기록 | `/activity` | 완료/실패/재설계 이력 확인 | 성과 순위, 감시형 리포트 |

MVP 1차 구현은 온보딩, 오늘의 퀘스트, 실패 이유 입력, 재설계 결과, 대시보드만 연결해도 충분하다. 이력서 점검, 모의면접, 공고 저장, 정책 추천은 퀘스트 category로 노출하고 독립 화면은 후순위로 둔다.

## 3. 핵심 사용자 흐름

1. 사용자가 온보딩에서 현재 상태를 입력한다.
2. 저장 후 오늘의 퀘스트 화면으로 이동한다.
3. 사용자가 `오늘의 퀘스트 생성`을 누른다.
4. 시스템은 10~30분 안에 끝낼 수 있는 퀘스트 3개를 카드로 보여준다.
5. 사용자는 각 카드에서 `완료` 또는 `어려웠어요`를 선택한다.
6. `어려웠어요`를 선택하면 실패 이유 입력 화면이 열린다.
7. 사용자는 이유를 선택하고 필요한 경우 한 줄 메모를 남긴다.
8. 시스템은 기존 퀘스트를 더 쉬운 행동 1~3개로 재설계한다.
9. 사용자가 재설계된 행동을 수락하면 오늘의 퀘스트 목록과 대시보드가 갱신된다.
10. 대시보드는 진행률보다 `다음에 할 수 있는 가장 작은 행동`을 먼저 보여준다.

## 4. 화면 상태 설계

| 화면 | 기본 상태 | 빈 상태 | 로딩 상태 | 오류 상태 |
|---|---|---|---|---|
| 온보딩 | 저장 가능한 form | 해당 없음 | 저장 버튼 비활성, 진행 표시 | 필드별 오류와 다시 시도 버튼 |
| 오늘의 퀘스트 | 퀘스트 카드 3개 | 생성 CTA와 온보딩 보강 안내 | skeleton 카드 3개 | 재시도 CTA, 기존 퀘스트 유지 |
| 실패 이유 입력 | 이유 선택 radio/listbox, 선택 메모 | 이유 미선택 시 제출 비활성 | 재설계 요청 중 버튼 비활성 | 이유 유지 후 다시 요청 |
| 재설계 결과 | 더 쉬운 행동 목록 | 재설계 후보 없음 안내와 직접 줄이기 CTA | 결과 skeleton | 원래 퀘스트를 더 작게 직접 수정하는 fallback |
| 대시보드 | 오늘 진행률, 다음 행동, 재설계 기록 | 오늘 퀘스트 생성 CTA | 요약 skeleton | 마지막 정상 데이터와 오류 배너 |

오류 상태는 사용자를 탓하지 않는 문구를 사용한다. 예를 들어 `요청을 처리하지 못했습니다. 지금 가능한 가장 작은 행동으로 다시 시도할 수 있어요.`처럼 행동 재진입을 돕는 표현을 쓴다.

## 5. 폼 검증 기준

### 온보딩

- `desiredJob`: 필수, 2~50자.
- `careerGapMonths`: 필수, 0 이상의 정수.
- `resumeStatus`: 필수, `NONE`, `DRAFT`, `READY` 중 하나.
- `interviewExperience`: 필수, `NONE`, `LOW`, `SOME` 중 하나.
- `energyLevel`: 필수, `LOW`, `MEDIUM`, `HIGH` 중 하나.
- `region`: 선택값. 정책 추천이 활성화되기 전까지는 필수로 만들지 않는다.

### 실패 이유 입력

- `failureReason`: 필수.
- 선택지는 사용자가 실패를 설명하기 쉬운 행동 중심 문구로 제공한다.
  - 시간이 부족했어요.
  - 할 일이 너무 크게 느껴졌어요.
  - 무엇을 써야 할지 몰랐어요.
  - 필요한 자료가 없었어요.
  - 오늘 컨디션이 낮았어요.
  - 공고가 맞지 않았어요.
- `memo`: 선택, 300자 이하.

## 6. 카피 가드레일

사용자에게 평가, 감시, 진단처럼 들리는 표현을 피한다.

| 피할 표현 | 대체 표현 |
|---|---|
| 의지 점수 | 오늘 가능한 에너지 |
| 실패 패턴 | 어려웠던 이유 |
| 위험 사용자 | 도움이 필요한 상황 |
| 성실도 | 실행 기록 |
| 진단 결과 | 오늘의 입력 요약 |
| 다시 실패했습니다 | 더 작은 행동으로 바꿔볼게요 |

CTA는 `완료`, `어려웠어요`, `더 작게 바꾸기`, `오늘 할 행동으로 저장`처럼 다음 행동이 분명한 문구를 사용한다.

## 7. API 계약 기대값

프론트엔드는 자연어 응답을 직접 파싱하지 않는다. 백엔드는 카드 렌더링과 상태 전환에 필요한 구조화된 JSON을 반환해야 한다.

```json
{
  "id": "quest-1",
  "title": "관심 공고 2개 저장하기",
  "description": "희망 직무와 가까운 공고를 2개만 찾아 저장합니다.",
  "category": "JOB_SEARCH",
  "difficulty": "EASY",
  "estimatedMinutes": 15,
  "status": "TODO",
  "completionCriteria": "공고 2개 저장",
  "fallbackQuest": {
    "title": "직무 키워드 1개만 검색하기",
    "estimatedMinutes": 5
  }
}
```

재설계 API는 원래 퀘스트, 실패 이유, 재설계된 행동 목록, 대시보드 반영 여부를 함께 반환하는 형태가 좋다.

## 8. 프론트엔드 책임 분리

React 구현 시 feature 단위로 나눈다.

```text
src/features/onboarding/
  pages/OnboardingPage.tsx
  components/OnboardingForm.tsx
  hooks/useOnboardingForm.ts
  types/onboarding.ts

src/features/quests/
  pages/TodayQuestPage.tsx
  components/QuestCard.tsx
  components/FailureReasonForm.tsx
  components/RedesignedQuestList.tsx
  hooks/useTodayQuests.ts
  hooks/useFailQuest.ts
  types/quest.ts

src/features/dashboard/
  pages/DashboardPage.tsx
  components/ProgressSummary.tsx
  components/NextActionPanel.tsx
  components/RedesignHistoryList.tsx
  hooks/useDashboard.ts
```

페이지 컴포넌트는 route 조립만 담당한다. API 호출, mutation, form state, view model 계산은 hook이나 service로 분리한다.

## 9. 접근성 및 반응형 기준

- 모바일 360px 폭에서 온보딩 form과 퀘스트 카드가 가로 스크롤 없이 보여야 한다.
- 퀘스트 카드의 주요 버튼은 최소 44px 높이를 확보한다.
- 실패 이유 선택지는 radio group 또는 접근 가능한 listbox로 구현한다.
- 재설계 결과가 생성되면 heading 또는 live region으로 상태 변화를 알려준다.
- 키보드만으로 `퀘스트 생성 -> 실패 이유 선택 -> 재설계 수락` 흐름을 완료할 수 있어야 한다.
- 색상만으로 완료/실패 상태를 구분하지 않고 label과 icon을 함께 사용한다.
- 대시보드는 데스크톱에서 2열까지 허용하되, `다음 행동` 영역이 첫 화면 상단에 남아야 한다.

## 10. 시각 검증 계획

현재 레포에는 실행 가능한 프론트엔드 앱이 없으므로 이번 문서 변경에는 스크린샷을 생성하지 않는다. 프론트 화면 골격이 추가되는 첫 구현 PR에서는 아래 화면을 desktop 1280px, mobile 390px 기준으로 캡처한다.

- 온보딩 입력 화면
- 오늘의 퀘스트 카드 3개가 있는 화면
- 실패 이유 입력 화면
- 재설계 결과 화면
- 대시보드의 다음 행동/재설계 기록 영역

시각 검증에서는 텍스트 겹침, 버튼 높이, mobile overflow, 평가성 문구 노출 여부를 함께 확인한다.

## 11. 사용자 관점 리스크

- 재설계 CTA가 숨겨지면 서비스가 일반 퀘스트 앱처럼 보일 수 있다.
- 대시보드가 통계 중심으로 치우치면 사용자의 다음 행동이 약해진다.
- 실패 이유 문구가 평가적으로 보이면 사용자가 이탈할 수 있다.
- 이력서, 면접, 공고, 정책 화면을 먼저 크게 만들면 MVP의 차별점이 흐려진다.
- API가 자연어 본문 중심이면 프론트 상태 관리와 접근성 안내가 불안정해진다.
